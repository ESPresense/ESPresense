# ESPresense positioning accuracy — public baseline

This is the first publicly tracked accuracy baseline for ESPresense. It is
**simulated**, not measured on hardware, and its purpose is to give the
project a number to defend and a number to regress against. Real-hardware
in-the-loop measurements will follow and will be published in the same
report format alongside this one — that work is tracked separately.

The harness lives at `tools/accuracy_harness/`. Re-running it on this
firmware revision should reproduce the numbers below byte-for-byte
(seeded RNG, see `--seed`).

## How to reproduce

```bash
python3 -m tools.accuracy_harness baseline \
  --output tools/accuracy_harness/reports/baseline-v1.json \
  --markdown tools/accuracy_harness/reports/baseline-v1.md
```

No external Python dependencies; standard library only. Default seed is
`20260426` to match the checked-in report.

## Scenario: tier-1 ESP32, 6m × 4m two-room residential

Four ESP32 nodes, ceiling-corner-mounted at 2.4m, in a 6m × 4m floor
divided into a 4m living room and a 2m bedroom. Two ground-truth tracks:

- `diagonal_walk`: corner-to-corner, 61 samples
- `living_to_bedroom`: a transit straight through the room boundary at
  x=4.0m, 121 samples — explicitly designed to exercise the failure mode
  reported in [#1817](https://github.com/ESPresense/ESPresense/issues/1817)
  (room-allocation flipping)

Noise profile applied to RSSI (dB) before the firmware's log-distance
formula converts to meters:

- Gaussian σ = **2.5 dB**
- Multipath bursts: **10%** of samples take a **-6 dB** attenuation hit

### What the numbers mean

- Median ~0.8m means most samples place the device within roughly a meter
  of where it actually is — about the size of a person.
- p95 ~2.5–3m means the worst 5% of samples land a couple of meters off,
  which on a 6m × 4m floor is enough to flip rooms.
- Room mis-id ~30% looks alarming on first read, but the
  `living_to_bedroom` track is *deliberately* a transit through the
  boundary — most of the misid samples are within a meter of the wall.
  The number is honest, not catastrophic. We will publish a separate
  "stationary in room center" track that should be far lower.
- Dropping any single station roughly doubles the worst-case error.
  Stations on the bedroom side hurt the most when removed because the
  bedroom-side geometry has fewer redundant constraints (only 2 of the 4
  stations are bedroom-adjacent).

## What this v1 does NOT model (blind spots)

These are deliberate, documented v1 limitations. Each is tracked as
follow-up work; none of them invalidate the baseline as a regression
guard for the firmware's own log-distance + smoothing path.

1. **No real RF.** Noise is injected as Gaussian RSSI jitter plus a
   single-channel multipath burst model. Real deployments see
   board-to-board variance, antenna pattern asymmetry, body absorption,
   and bluetooth-channel-dependent fading that this harness ignores.

2. **No per-board RSSI variance.** Every station uses the same
   `cal_rssi = -59 dBm` and `absorption = 3.0`. Tier-1 hardware variants
   (ESP32, ESP32-C3, ESP32-S3, ESP32-C6) have measurable receive-side
   differences. Per-board calibration models will be added when we have
   on-hardware capture data to fit them against. The
   distance-calc fixture suite (`test/test_native_distance/`) covers the
   formula itself per board, but RSSI input distributions are still
   firmware-uniform here.

3. **Reference solver, not Companion's actual `ILocate`.** The harness
   uses a pure-Python Nelder-Mead minimizing
   `Σ (||p − stationᵢ|| − dᵢ)²`, which is the same residual form
   Companion publishes in
   [`NelderMeadMultilateralizer.cs`](https://github.com/ESPresense/ESPresense-companion/blob/main/src/Locators/NelderMeadMultilateralizer.cs).
   It does NOT include Companion's optional `scale` parameter, weighting
   schemes (Gaussian/inverse), or the `Pearson` confidence pruning.
   These all *help* accuracy in real deployments, so the published
   numbers are a **lower bound** on Companion's true behavior. A
   Companion-side issue is filed to expose `ILocate` as a callable
   library / CLI; once that ships, this harness will swap solver and we
   will re-baseline. The stub is documented as such in
   [`tools/accuracy_harness/solver.py`](../tools/accuracy_harness/solver.py).

4. **2D positioning only.** Z is fixed at the station-z plane in v1. The
   firmware and Companion both solve in 3D; ours doesn't. Most
   residential layouts place all nodes within ~1m of each other
   vertically, so the projection error is small but not zero.

5. **No Kalman / hysteresis.** Companion runs a Kalman filter on the
   solver output that smooths and rejects physically-impossible jumps.
   Our harness reports the raw per-sample solve, which is the right
   layer for catching firmware-side regressions but is noisier than
   what an end user sees.

6. **Single-floor, axis-aligned rooms.** Multi-floor and polygon-shaped
   rooms (also supported by Companion) are out of scope for v1.

## How to add a scenario

Drop a JSON file into `tools/accuracy_harness/scenarios/` in the same
shape as `baseline_4node_6x4m.json`, then run

```bash
python3 -m tools.accuracy_harness baseline --scenarios your_scenario.json
```

Open a PR with the JSON file, the noise profile you used, and the
checked-in report. We accept community-submitted scenarios.

## Regressions

If a firmware change moves a checked-in baseline number by more than ~5%
on the `baseline_4node_6x4m` scenario, the PR description must explain
why. The current baseline JSON is at
[`tools/accuracy_harness/reports/baseline-v1.json`](../tools/accuracy_harness/reports/baseline-v1.json).
